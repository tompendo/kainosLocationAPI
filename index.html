Why these two flows exist

Flow A — Kainos-defined check types (e.g. Business Cycle Conflicts) have their category in the browser URL. Pendo can read this directly. addTransforms only needs to append which tab the user is on — the check type context is already there.

Flow B — Customer-defined check types all share the same URL structure (#/audit/checkDetails/221). The numeric ID changes per check but gives Pendo no context about module or check type — SOD, Privileged Access and Config Change all look identical. addTransforms must inject the check type identity at page load — context is available at that moment per Eamon's confirmation — and then update the tab state on each tab click.
